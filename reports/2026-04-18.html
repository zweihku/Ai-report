模型与算法
算法 Google TurboQuant：KV Cache 内存压缩的重大突破

Google 在 ICLR 2026 上发布 TurboQuant，专门针对 Transformer 模型中 KV cache 这一最大内存瓶颈。传统方法中，KV cache 的内存占用会随上下文长度线性增长，对于 GPT-6 级别的 200 万 token 上下文窗口，KV cache 单独就需要数十 GB 显存。

TurboQuant 通过动态量化和稀疏注意力模式相结合，将内存需求降低 6 倍，同时保持推理质量损失低于 0.5%。这意味着原本需要 A100 80GB 的推理任务，现在可以在 A100 40GB 甚至消费级 GPU 上运行。对于 API 提供商来说，同等硬件的吞吐量可提升 4-5 倍。

核心技术分析
算法 TurboQuant 深度解析：动态量化 + 稀疏注意力的 KV Cache 革命

KV Cache 是 Transformer 自回归推理中最大的内存瓶颈。对于一个 70B 参数模型在 128K context 下推理，KV cache 就需要约 40GB 显存。GPT-6 的 200 万 token 上下文窗口将这一数字推到了不可接受的水平。

TurboQuant 的技术方案包含三个核心创新：

  • Layer-Adaptive Quantization：不同 Transformer 层的 KV cache 使用不同的量化精度（FP16/INT8/INT4），基于每层 attention entropy 动态决定精度分配
  • Sliding Window Sparse Attention：对超出滑动窗口的远距离 token，仅保留 top-k 重要度的 KV 对，其余丢弃而非量化
  • Cross-Layer KV Sharing：相邻层共享部分 KV cache，利用 Transformer 层间表示的相似性减少冗余存储

最终效果：内存降低 6 倍，推理速度提升约 2.3 倍，输出质量损失 < 0.5%（MMLU、HumanEval、MATH 上测量）。这一成果对 long-context 推理场景的商业化部署具有决定性意义。

国家科技信息中心
安全 反模型蒸馏系统的技术架构分析

三大实验室联建的反蒸馏系统在技术架构上涉及多个层面：

检测层

  • Query Pattern Analysis：基于 n-gram 和 embedding 相似度检测系统性的 prompt 覆盖行为。正常用户的查询分布呈长尾分布，而蒸馏行为的查询分布趋向于均匀覆盖模型能力空间
  • Response Fingerprinting：在模型输出中嵌入统计层面的隐性水印（非文本水印），用于验证下游模型是否使用了蒸馏数据
  • Behavioral Clustering：使用无监督学习对 API 调用模式进行聚类，识别大规模自动化调用的特征指纹

响应层

  • Coordinated Rate Limiting：跨平台共享可疑账号特征后，三家同步实施限流
  • Quality Degradation：对确认的蒸馏行为渐进降低输出质量，而非直接封禁，增加对手的检测难度
  • Legal Escalation：收集证据链支持法律行动，Anthropic 已向美国国会提交相关材料

OpenAI 向国会提交的材料指控中国企业通过混淆访问路径批量获取模型输出。Anthropic 公开指控涉及数千假账号和超 1600 万次交互。这一攻防战的技术水平已远超普通的 API abuse 防护。

架构 MoT vs MoE：腾讯 HY-Embodied 的架构选择

腾讯 HY-Embodied-0.5 选择 Mixture-of-Transformers（MoT）而非主流的 Mixture-of-Experts（MoE）架构，这一选择值得深入分析：

  • MoE 在 FFN 层面做专家化：不同 expert 处理不同类型的 token，routing 发生在 FFN 之前。优势是计算效率高（稀疏激活），但不同 expert 共享相同的 attention 表示
  • MoT 在 Transformer 层面做专家化：不同 expert 拥有独立的 attention 机制。优势是不同 expert 可以学习不同的 attention pattern，对多模态输入（视觉、触觉、语言）的特化处理更优
  • 在具身 AI 场景中，机器人需要同时处理视觉流（30fps+）、触觉反馈（100Hz+）和语言指令（异步），MoT 的模态级专家化比 MoE 的 token 级专家化更自然
